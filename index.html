ECMAScript.next
History of ECMAScript.next

* ECMAScript 3 (Dec 1999)

* ECMAScript 4 (2008)

* ECMAScript 3.1

* ECMAScript 3.1 === ECMAScript 5 (Dec 2009)
* ECMAScript 4 ECMAScript Harmony

Why not Harmony?

* Harmony includes too much ideas
* ES.next === subset of Harmony features
* ES.next === ECMAScript 6 (almost certainly)

let

* let is like var with block scope
* avoids "hoisting" errors

const

* like "final" in Java
* now officially in spec
* and with block scope

Modules

* no more (function(){})();

module Something {
    // Internal
    var internalId = 123456;
    // External
    export function doSomething() {
        //
    }
    export var somePublicVar = "Public";
};
Modules

* all exports can be imported 

import doSomething from Something;
import Something as ST;
import 'http://json.org/modules/json2.js' as JSON;
Classes

* just syntactic sugar (for prototype chaining)
* easier to read
* less typing

Default Parameter Values

* no more options = options || {}; 

function doSomething(val = 'Do something') {
    console.log(val);
}
doSomething() // prints 'Do something'
Rest parameter

* like Java Varargs 

function f(a, b, ...args) { 
    console.debug(Array.isArray(args));
    return args.concat(a, b); }
Object.observe()

* observes modifications of an object 

var obj = {};

Object.observe(obj, function(changes) {
    changes.forEach(function(change, i) {
        console.log(change);
    });
});
Sets & Maps

* use of non-string keys
* support for iteration over keys & values
* no concern over prototypes

WeakMaps

* like Java (remove values when key object gets garbage collected)
* only objects as keys allowed
* use case: store values for DOM elements

for-of iteration

* iterate over value
* used with new iterator factory feature
